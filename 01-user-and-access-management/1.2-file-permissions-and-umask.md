## 🔢 Core Permission Triplets

Linux divides file system interactions into three operational classifications: **Read (r)**, **Write (w)**, and **Execute (x)**.

| Permission Type | Symbolic Value | Octal/Numeric Value |
| :--- | :---: | :---: |
| **Read** | `r` | 4 |
| **Write** | `w` | 2 |
| **Execute** | `x` | 1 |

### Targeted Scopes
* **`u`** : User (Owner of the asset)
* **`g`** : Group (Members assigned to the asset's group)
* **`o`** : Others (Everyone else on the system)
* **`a`** : All (Applies a rule to user, group, and others simultaneously)

---

## 🛠️ Modifying File Access Modes (`chmod`)

### 1. Numeric/Octal Assignment
Sets absolute permission bits directly.

# Sets User=rwx (7), Group=rw- (6), Others=--- (0)
chmod 760 test_file

```

### 2. Symbolic Assignment

Appends, removes, or forces specific relative modes. Use the `-v` flag to explicitly print a detailed output log of the structural shifts made.

```bash
# Append Read and Write capabilities to the group
chmod -v g+rw test_file

# Revoke Write and Execute capabilities from both Users and Others
chmod -v uo-wx test_file

# Erase all access metrics completely for Users and Others
chmod -v uo= test_file

```

### 📁 Directory Level Rules

```bash
# Modify access targeting an explicit folder block
chmod u+rwx target_folder

# Recursively application (-R): Modifies the folder and every single file inside it
chmod -R g+rw target_folder/

```

## 👑 Modifying Ownership (`chown` / `chgrp`)

```bash
# Alter the primary owning group allocation explicitly
chgrp developers target_asset

# Transition account owner status over to a new user
chown alex target_asset

# Transition both Account Owner AND Group simultaneously using a colon `:`
chown alex:developers target_asset

# Transition owner and automatically match their primary group using a dot `.`
chown alex. target_asset

```

## 🎭 Default File Creation Mask (`umask`)

The system `umask` acts as a filter that strips away permissions when a new file or directory is generated.

* Base system state for a brand new **Directory**: `777`
* Base system state for a brand new **Regular File**: `666`

$$\text{Final Permissions} = \text{Base State} - \text{Umask Value}$$

### Umask Comparison Table

| Set Umask Value | Calculation Formula | Directory Outcome | File Outcome |
| --- | --- | --- | --- |
| `umask 022` | Directory: $777 - 022 = 755$ <br>

<br> File: $666 - 022 = 644$ | `rwxr-xr-x` (Standard) | `rw-r--r--` (Standard) |
| `umask 000` | Directory: $777 - 000 = 777$ <br>

<br> File: $666 - 000 = 666$ | `rwxrwxrwx` (Unsafe) | `rw-rw-rw-` (Unsafe) |
| `umask 037` | Directory: $777 - 037 = 740$ <br>

<br> File: $666 - 037 = 620$ | `rwxr-blank-blank` | `rw-r--blank-blank` |

```bash
# Check or configure active process mask configuration dynamically
umask 022

```

## ⚡ Special System Permissions (SUID, SGID, Sticky Bit)

When standard user blocks aren't flexible enough for shared workspaces or elevated execution, utilize the 4th high-order octal prefix digit:

```bash
# Syntax layout structure
chmod <SpecialBit><StandardBits> target_folder

```

### The Three Special Operations

1. **SUID (Set Owner User ID) — Octal `4**`: Processes execute using the context privileges of the file's owner rather than the user invoking it.
2. **SGID (Set Group ID) — Octal `2**`: New assets created within this directory automatically inherit the folder's parent group ID, rather than the creating user's primary group.
3. **Sticky Bit — Octal `1**`: Restricts deletion loops. Only the explicit file owner or root can rename or delete a file inside this workspace.

> 💡 **Best Operational Practice (Value `3`)**: Combining SGID (`2`) and Sticky Bit (`1`) equals **`3`**. This ensures group members can freely modify files inside a shared workspace, but cannot delete or hijack files belonging to other team members.

```bash
# Example: Enforce SGID, Sticky Bit, and rwxrwxr-x on a workspace directory
chmod 3775 shared_workspace/

```
